## Phase 3: Build the MVP

### Goal

Create the simplest version of the platform that helps motivated beginners move from **“I want an internship, but I don’t know where to start”** to **“I completed a career-relevant team project and can use it for applications.”**

The MVP should not be a full social network or job board.
It should be a guided project platform that helps students understand their career goal, find teammates, follow milestones, and create a final output for resumes, portfolios, GitHub, LinkedIn, or interviews.

---

## MVP Thesis

The strongest MVP is not just a matching platform.

It should be a **career-readiness project accelerator** for early undergraduate students.

The platform should help users answer four questions:

1. What career path am I interested in?
2. What experience do I need for that career?
3. What kind of project should I complete?
4. How can I turn that project into career value?

---

## Target Users for MVP

The MVP should focus on motivated freshman to junior college students in the United States who want internships but lack practical project experience.

### Best-fit users

* Students interested in software engineering
* Students interested in data science or data analytics
* Students interested in product management
* Students interested in UX or product design
* Students interested in marketing or growth
* Students interested in consulting, startups, or business strategy

These users benefit most because project-based experience can become visible proof of skills through:

* GitHub repositories
* Product documents
* Strategy decks
* UX case studies
* Data dashboards
* Marketing campaigns
* Resume bullet points
* LinkedIn project descriptions
* Interview stories

## Core MVP Features

| Feature              | Purpose                                                                                                                 |
| -------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| Landing Page         | Explain who the platform is for, what users can build, and how the process works                                        |
| User Profile         | Collect school, year, major, skills, interests, career goals, availability, and links                                   |
| Career Track Pages   | Explain different roles, required skills, useful project examples, and expected outputs                                 |
| Role Guide           | Show what each role does in a team project and what responsibilities they have                                          |
| Team Matching        | Match users into small teams based on interests, role, skill level, and availability                                    |
| Project Tracks       | Provide beginner-friendly project options by career path                                                                |
| Weekly Milestones    | Guide teams step by step from project kickoff to final deliverable                                                      |
| Project Workspace    | Give each team a place to see tasks, roles, deadlines, and deliverables                                                 |
| Final Output Export  | Help users turn their project into a resume bullet, portfolio summary, GitHub README, LinkedIn post, or interview story |
| Trust / Verification | Use school email, profile links, and conduct guidelines to make the community safer and more serious                    |

## Career Track Pages

Career Track Pages are important because beginners often do not know what experiences are useful for their target careers.

Each career page should explain:

* What this career path is
* Common internship titles
* What employers look for
* Why project experience matters
* What skills students should build
* What beginner-friendly projects they can complete
* What final output they should create

### Example career tracks

| Career Track                   | Useful Project Output                                       |
| ------------------------------ | ----------------------------------------------------------- |
| Software Engineering           | Web app, mobile app, API, GitHub repo, demo video           |
| Data Science / Analytics       | Dashboard, notebook, analysis report, visualization, model  |
| Product Management             | PRD, user research, roadmap, feature proposal, product deck |
| UX / Product Design            | Figma prototype, UX case study, user interview summary      |
| Marketing / Growth             | Campaign plan, content calendar, landing page, KPI report   |
| Consulting / Business Strategy | Market analysis, strategy deck, recommendation memo         |
| Startup / Entrepreneurship     | MVP, pitch deck, customer discovery report                  |

## Role Guide

The Role Guide helps beginners understand what they can contribute to a team.

Each role page should include:

* Plain-English explanation of the role
* Weekly responsibilities
* Tools commonly used
* What “good work” looks like
* How this role connects to real jobs
* Resume bullet examples
* Common beginner mistakes

### Example roles

| Role                   | Main Responsibility                                                                         |
| ---------------------- | ------------------------------------------------------------------------------------------- |
| Product Manager        | Defines the problem, organizes user needs, prioritizes features, and keeps the team aligned |
| Developer / Engineer   | Builds the product, website, app, backend, or technical prototype                           |
| Designer / UX Designer | Creates user flows, wireframes, prototypes, and improves the user experience                |
| Data Analyst           | Cleans data, analyzes patterns, builds dashboards, and explains insights                    |
| Business Strategist    | Researches the market, analyzes competitors, and creates business recommendations           |
| Marketing Lead         | Plans campaigns, creates content, tests messaging, and measures results                     |

## Matching System

The MVP matching system does not need to be fully automated at first.

For the first pilot, manual or semi-manual matching is enough.

### Matching criteria

* Career interest
* Preferred role
* Skill level
* Project interest
* Weekly availability
* Time zone
* Seriousness / commitment level
* Team balance

### Ideal team size

3–5 people per team.

### Example team structure

```md
1 Product Manager
1 Developer
1 Designer
1 Business / Marketing / Data person
```

---

## Project Workspace

Each team should have a simple workspace that shows:

* Team members
* Roles
* Project goal
* Weekly milestones
* Tasks
* Deadlines
* Deliverables
* Uploaded artifacts
* Final output checklist

The workspace should always make three things clear:

1. What is my role?
2. What should I do this week?
3. What final output are we trying to create?

---

## Weekly Milestone System

Each project should follow a simple 4–6 week structure.

### Example 5-week project cycle

| Week   | Goal                          | Output                                                          |
| ------ | ----------------------------- | --------------------------------------------------------------- |
| Week 1 | Define problem and team roles | Problem statement, role assignment                              |
| Week 2 | Research and plan             | User research, market research, wireframe, or technical plan    |
| Week 3 | Build or analyze              | Prototype, dashboard, code, campaign, or strategy draft         |
| Week 4 | Improve and test              | Feedback, revisions, usability test, QA, or analysis refinement |
| Week 5 | Finalize and present          | Final project, resume bullet, portfolio summary, presentation   |

---

## Trust and Verification

Because this platform involves teamwork, trust is important from the beginning.

### MVP trust features

* School email verification
* LinkedIn, GitHub, or portfolio link
* Clear community guidelines
* Availability and commitment check
* Report or feedback form
* Team participation expectations

The MVP should not require government ID or sensitive personal information at the beginning.
